# JMeter Scalability Baseline Test

This folder is for experiment 4 only: scalability testing of the original, unrefactored project.

The experiment 3 performance baseline can use the frontend CSV produced by the page operation logs. This JMeter plan focuses on how the current system behaves as concurrent users increase.

## What This Test Covers

The test plan intentionally avoids Excel upload and `save-final-to-db`.

- Excel upload is already covered by the experiment 3 CSV baseline.
- `save-final-to-db` is a write operation with `overwrite=true`; running it concurrently can repeatedly delete and insert the same result set, which is noisy and not suitable for a clean scalability baseline.

The JMeter plan tests these read/compute/export endpoints:

- `GET /api/data-status`
- `GET /api/final-rows-db?limit=100`
- `POST /api/compose-final`
- `POST /api/generate-final-excel`

These endpoints represent light query, saved result query, core composition logic, and Excel export pressure.

## Prerequisites

1. Start MySQL and make sure the project database contains the full imported data.
2. Start the backend with `AcademicApplication`.
3. Make sure the final report rows have already been saved from the frontend page, so `/api/final-rows-db?limit=100` has data.
4. Install Apache JMeter and make `jmeter` available in PATH, or pass the full path to `jmeter.bat`.

## Run

From the project root:

```powershell
powershell -ExecutionPolicy Bypass -File .\tests\jmeter\run-scalability-baseline.ps1
```

If JMeter is not in PATH:

```powershell
powershell -ExecutionPolicy Bypass -File .\tests\jmeter\run-scalability-baseline.ps1 -JMeterBin "D:\tools\apache-jmeter-5.6.3\bin\jmeter.bat"
```

Default concurrency levels:

- 1 users
- 5 users
- 10 users
- 30 users
- 50 users

Each user runs the endpoint sequence 5 times.

## Output

Results are generated under:

```text
tests/jmeter/results/baseline-YYYYMMDD-HHMMSS/
```

Each concurrency level contains:

- `users-N.jtl`: raw JMeter result data
- `users-N-html/`: HTML dashboard generated by JMeter

For the Word report, take screenshots from each HTML dashboard or from JMeter Aggregate Report:

- Average response time
- 90% Line / 95% Line
- Throughput
- Error %

## Report Interpretation

Scalability is evaluated by whether response time grows smoothly as concurrency increases, whether throughput rises with more users, and whether error rate remains close to 0%.

If response time rises sharply, throughput stops growing, or errors appear at higher concurrency, the current architecture has scalability bottlenecks. For this project, likely bottlenecks include synchronous Excel export, single Spring Boot instance, direct database connection creation in `DBUtil`, and the lack of asynchronous task processing.
